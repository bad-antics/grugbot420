What is bindboss?

bindboss takes a directory full of code, data, scripts — whatever you want — and packages it into a single self-extracting binary. When someone runs that binary, it unpacks itself into a temp directory, optionally checks and installs dependencies, then executes your run command. The end user never interacts with bindboss directly.

It was built to solve the "Julia is painful to bundle" problem, but it works for any runtime: Python, Bun, Node, Ruby, shell scripts, compiled apps that need data files alongside them — anything.

📚 Academic
bindboss implements a self-contained executable distribution pattern: a statically compiled stub binary acts as a runtime loader, with the application payload appended as a tar.gz archive. At execution time, the stub extracts the payload to a temporary or persistent directory and execve(2)s the configured run command, replacing itself with the target process. This eliminates the traditional extraction/installer step entirely from the user's perspective.

What it is NOT

  • Not a container runtime (no kernel namespacing, no image layers)
  • Not a package manager (it doesn't version-resolve deps)
  • Not a compiler or bundler (it doesn't compile your code)
  • Not a virtual machine or interpreter

It is a packer. It puts your files inside a binary. The binary unpacks them and runs your command. That's the whole thing.

Installation

Option A: Go install (easiest)

shell
go install github.com/marshalldavidson61-arch/bindboss@latest

This compiles bindboss and puts it in your $GOPATH/bin. Requires Go 1.22.3 or later.

Option B: Build from source

shell
git clone https://github.com/marshalldavidson61-arch/Bindboss
cd Bindboss
go build -o bindboss .
sudo mv bindboss /usr/local/bin/    # optional: make it available system-wide

Option C: Use the pre-built binary

A pre-built binary is included in the repository at the root level. Download it and make it executable:

shell
curl -L https://github.com/marshalldavidson61-arch/Bindboss/raw/main/bindboss -o bindboss
chmod +x bindboss
./bindboss pack --help
✓ Tip
When bindboss runs pack, it needs to compile the stub binary using go build. So even if you use Option C, Go must be installed on the machine doing the packing. The end user running the packed binary does NOT need Go.

Verify installation

shell
bindboss

You should see the command listing. If you get "command not found", make sure your $GOPATH/bin is in your $PATH.

How it works

Understanding the internals helps you understand why certain things work the way they do — especially around file paths in your run command and how deps are checked.

The packing pipeline

When you run bindboss pack, three things happen:

  1. Stub compilation — bindboss compiles a tiny Go binary called the "stub" using go build -tags stub. This stub is statically linked (no libc). It contains all the extraction and execution logic.
  2. Staging — your source directory is copied to a temp dir, and bindboss.toml (your config, merged with CLI flags) is written into it.
  3. Payload append — the staged directory is compressed as tar.gz and appended to the stub binary. A 121-byte v2 trailer is written after it containing the tar offset, SHA-256 hash, Ed25519 signature slot, and magic bytes.
📚 Academic — v2 Trailer Layout
The payload trailer is exactly 121 bytes appended to the end of the binary:

[8 bytes big-endian uint64 tar_offset ]
[32 bytes SHA-256 of raw tar.gz bytes ]
[64 bytes Ed25519 signature OR zeros ]
[1 byte flags: bit0=hash_present, bit1=sig_present]
[16 bytes magic: "BINDBOSS_PAYLOAD\x02" ]

The stub locates the trailer by reading the last 16 bytes to find the magic, then seeks backward to read the full trailer. This append-only structure means the stub ELF/PE/Mach-O header is never modified — only data is appended — preserving binary validity.

What the stub does at runtime

When you run a packed binary, the stub executes in this order:

  1. Find itself on disk (resolves symlinks)
  2. Locate the payload trailer and open the tar.gz reader
  3. If BINDBOSS_VERIFY=1, re-read and verify SHA-256 before proceeding
  4. Determine the extract directory (temp or persist)
  5. Extract the tar.gz payload into that directory (skipped if persist dir already has content)
  6. Load bindboss.toml from the extracted directory
  7. On first run: run the install wizard (if configured) or check deps the old way
  8. Write dep-checked state so step 7 is skipped on subsequent runs
  9. Set extra environment variables from config
  10. Run pre_run hooks
  11. execve(2) the run command — the stub process is replaced entirely
  12. (Fork mode only) run post_run hooks, then cleanup

File paths in your run command

The stub chdirs into the extracted directory before running your command. So your run command's relative paths are relative to the root of your packed directory.

Example 1 Understanding path resolution

├── src/
│   └── main.py       ← this is what gets run

bindboss.toml

If you put a bindboss.toml file inside your directory before packing, bindboss reads it and uses it as the config. CLI flags override anything in the file. You can also use only CLI flags — the TOML file is optional.

The TOML is injected into the packed binary at pack time. The stub always reads it from the extracted directory at runtime. You never need to ship the TOML separately.

Full annotated example

tomlbindboss.toml
# Display name — used in logs, state file path, and wizard header
name = "myapp"

# Command to run inside the extracted directory
# Relative paths are relative to the packed directory root
run = "python3 src/main.py"

# "exec" (default) or "fork"
# exec = syscall.Exec, replaces process, no post_run, clean PID
# fork = subprocess, stub stays alive, post_run hooks work
exec_mode = "exec"

# Extra environment variables injected before running
env = ["APP_ENV=production", "LOG_LEVEL=info"]

# Runtime dependencies checked on first run
# Without the install wizard, missing deps open a browser to the URL
[[needs]]
name    = "python3"
check   = "python3 --version"
url     = "https://python.org/downloads/"
message = "Python 3.8+ required. Download from python.org."

# Hooks: commands run before/after the main run command
# Each string is argv — no shell expansion, no $VAR substitution
# Use ["sh", "-c", "your shell cmd here"] for shell features
[hooks]
pre_run  = ["python3 -m pip install -r requirements.txt -q"]
post_run = []  # only fires in exec_mode = "fork"

# Extraction settings
[extract]
persist = true   # reuse fixed dir (~/.bindboss/myapp/) across runs
cleanup = false  # don't delete on exit (ignored when persist=true)

# Install wizard (optional — replaces the basic dep check on first run)
[install]
enabled      = true
install_file = "install.json"   # path relative to packed dir root

Configuration precedence

bindboss uses a two-layer merge: TOML values are defaults, CLI flags override them. The merged config is written back into the packed binary — so the stub always sees a single authoritative bindboss.toml.

shell
# toml says run="python3 main.py", but --run flag overrides it
bindboss pack ./myapp myapp-bin --run="python3 main_v2.py"

# --persist flag overrides extract.persist in TOML
bindboss pack ./myapp myapp-bin --persist
📚 Academic
The merge strategy is implemented in config.MergeFlags(). Empty string from CLI means "not provided" and preserves the TOML value. This is deliberate: it prevents a flag's zero value from silently clobbering a TOML setting. The merged config is serialized back to TOML via config.ToTOML() using BurntSushi/toml and embedded in the staged directory before archiving.

Payload format

Packed binaries use the v2 format. Older binaries from before v0.2.0 use v1 (24-byte trailer, no hash, no sig). The stub transparently reads both.

FormatTrailer sizeHashSignature
v1 (legacy)24 bytesNoNo
v2 (current)121 bytesSHA-256Ed25519 (optional)

To check which format a binary uses:

shell
bindboss inspect ./myapp   # shows "format: v1" or "format: v2"

To upgrade a v1 binary to v2 (get hash verification), simply repack it with the new bindboss.

Command: pack

bindboss pack [flags]

Packs a directory into a self-extracting binary. This is the command you'll use 95% of the time.

Flags

FlagRequiredDescription
--run="cmd"required*Command to run inside the extracted dir. Required unless bindboss.toml already has a run field.
--needs="name,check,url"optionalDep to check on first run. Repeatable. Format: "name,checkCmd,url" or "name,checkCmd,url,message"
--persistoptionalExtract to fixed dir (~/.bindboss//) and reuse across runs. Faster for big runtimes.
--dir="path"optionalOverride the extract root directory.
--target="GOOS/GOARCH"optionalCross-compile for another platform: linux/amd64, darwin/arm64, windows/amd64, etc.
--sign="path/to/key"optionalSign the payload with an Ed25519 private key (generated with bindboss keygen).

Important: what gets packed

bindboss packs everything in the source directory, including hidden files and subdirectories. It does NOT automatically exclude .git/, node_modules/, build artifacts, etc. If you don't want something in the binary, don't have it in the source directory when you pack.

✓ Best practice
Use a clean staging directory for packing. Copy only what you need into a temp dir, then pack that. This keeps binary size down and avoids leaking .git/ history or dev credentials into your shipped binary.

mkdir -p /tmp/myapp-stage && rsync -a --exclude='.git' --exclude='*.pyc' ./myapp/ /tmp/myapp-stage/
bindboss pack /tmp/myapp-stage myapp-bin --run="python3 main.py"

Command: verify

bindboss verify [--pubkey=path]

Verifies the SHA-256 hash of the payload (always) and optionally the Ed25519 signature. Returns exit code 0 on success, 1 on failure.

Example 8 Verify integrity (hash only) 
bindboss verify ./grugbot420
Output
[bindboss] ✓ hash verified: 83aee33c...abd6db2e2
Example 9 Verify with signature check 
bindboss verify ./grugbot420 --pubkey=~/.bindboss/keys/myproject.pub
Success
[bindboss] ✓ hash verified: 83aee33c...
[bindboss] ✓ signature verified (Ed25519)

You can also trigger hash verification at runtime by setting an env var before running the binary:

shell
BINDBOSS_VERIFY=1 ./grugbot420

This forces the stub to re-read and verify the payload hash before extracting. It's off by default because it requires a full extra read of the payload (slower for large binaries).

Command: keygen

bindboss keygen [--keydir=path]

Generates an Ed25519 keypair for signing packed binaries. Creates two files: .key (private, mode 0600) and .pub (public). Keys are stored in ~/.bindboss/keys/ by default.

Example 10 Generate a keypair 
bindboss keygen myproject
Output
[bindboss] ✓ keypair generated: myproject
[bindboss]   private: /home/user/.bindboss/keys/myproject.key (0600)
[bindboss]   public:  /home/user/.bindboss/keys/myproject.pub
⚠ Warning
Keep your .key file private. Anyone with your private key can sign binaries that verify against your public key. The .pub file is safe to share — distribute it alongside your binary so users can verify authenticity.

Command: reset

bindboss reset

Deletes the first-run state for a binary. This causes the dep check or install wizard to run again on next launch. Useful for re-testing your installer or when a user reports a missing dep issue.

Example 11 Re-trigger the install wizard 
bindboss reset grugbot420
./grugbot420    # install wizard runs again

The state file is stored at ~/.bindboss/state/.json. Deleting that file manually has the same effect as bindboss reset.

Install Wizard — Overview

The install wizard is a JSON-driven terminal GUI that runs on first launch instead of the basic dep check. It provides a guided, multi-step setup experience — like a proper installer, but in the terminal, with no ncurses and no third-party dependencies.

welcome → license → deps → message → finish

You define the wizard entirely in JSON. The stub reads it on first run and executes each step in order. Steps can be skipped back to with the "back" command. The wizard only runs once — after it completes, state is written so subsequent runs go straight to your app.

🦕 Grug says
Grug make JSON file. JSON file say what wizard do. Wizard run on first launch. User see nice terminal boxes. User press Enter. User get through wizard. App run. Grug not write any Go code for this. Grug just write JSON. Very good.
📚 Academic
The wizard follows a discriminated union pattern in its JSON schema — each step object has a "type" field that acts as a type discriminator. The parser rejects unknown types at startup (fail-fast), preventing mid-install failures. The execution model is a simple finite state machine: a currentStep index advances forward on Enter and backward on "back". All terminal rendering uses Unicode box-drawing characters (U+2550 et al.) with no ANSI cursor movement — output is strictly append-only, making it pipeable and debuggable.

How to enable the wizard

Three steps:

  1. Create an install.json file inside your app directory
  2. Add the [install] section to your bindboss.toml
  3. Pack as usual
tomlbindboss.toml
[install]
enabled      = true
install_file = "install.json"  # relative to packed dir root

If enabled = false or the [install] section is absent entirely, bindboss falls back to the original behavior: check deps → open browser if missing → wait for Enter → re-check. Existing binaries are completely unaffected.

Install Wizard — Dep Definitions

Dep definitions live inside a deps step. Each object defines one dependency the wizard will check and potentially download.

FieldRequiredDescription
namerequiredHuman-readable name shown in the UI, e.g. "Julia 1.9+"
checkrequiredShell command to test presence. Exit 0 = installed, non-zero = missing.
download_urlrequired*Direct HTTP(S) download link for the installer file.
fallback_urlrequired*Browser URL shown if direct download fails or isn't available.
file_nameoptionalOverride filename for the downloaded installer. Defaults to the URL's basename.
hashoptionalExpected SHA-256 hex digest of the downloaded file. If provided, download is rejected if hash doesn't match.
messageoptionalExtra context shown to the user above the download prompt.

* At least one of download_url or fallback_url must be provided.

Download flow per dep

For each dep in a deps step, the wizard does this:

  1. Run the check command. If exit 0 → dep is present, show ✓ and move on.
  2. If missing: show the dep name, message, and download prompt.
  3. HTTP GET the download_url with a progress bar.
  4. If hash is set: verify SHA-256 of downloaded file. Abort if mismatch.
  5. Launch the downloaded installer (platform-aware: .sh, .exe, .msi, .dmg, etc.).
  6. Prompt user to press Enter when installation is complete.
  7. Run check again. If still missing: offer retry. If present: show ✓.

Install Wizard — Navigation

InputAction
EnterProceed to next step
back + EnterGo back one step (where applicable)
retry + EnterRe-download a failed dependency
accept + EnterAccept the license agreement (license step only)
Ctrl+CAbort wizard entirely — exits with FATAL error, no state written

Navigation is case-insensitive. Pressing Enter on a non-license step always advances forward. Going back on the first step does nothing.

✓ Tip
If a user aborts with Ctrl+C, no state is written — the wizard will re-run on the next launch. This is intentional: a crashed or incomplete install should not be silently skipped next time.

Install Wizard — Inline vs File Config

You have two ways to provide the wizard JSON to bindboss:

Option 1: Reference a JSON file in your packed directory (recommended)

tomlbindboss.toml
[install]
enabled      = true
install_file = "install.json"   # path relative to packed dir root

The install.json file must exist inside the directory you're packing. The path is relative to the root of the packed directory. So if your app lives at myapp/src/ and you pack myapp/, use "src/install.json".

Option 2: Inline the JSON directly in TOML

tomlbindboss.toml
[install]
enabled        = true
install_config = '{"title":"My App","version":"1.0.0","steps":[{"type":"welcome","title":"Hi","content":"Hello!"},{"type":"finish","title":"Done","content":"Starting..."}]}'

Useful for simple configs or when you don't want a separate file. For anything with more than two steps, the file approach is much easier to read and edit.

⚠ Precedence
If both install_config and install_file are set, install_config (inline) wins. The file is ignored.

Advanced: Pre/Post Hooks

Hooks are commands that run before or after your main run command. They're defined as an array of strings in bindboss.toml. Each string is parsed as argv — no shell expansion, no variable substitution.

toml
[hooks]
pre_run  = [
    "python3 -m pip install -r requirements.txt -q",
    "python3 scripts/init_db.py"
]
post_run = ["python3 scripts/cleanup.py"]  # exec_mode=fork ONLY
⚠ post_run and exec mode
post_run hooks only fire in exec_mode = "fork". In the default exec mode, syscall.Exec replaces the stub process entirely — there's nothing left to run post_run after your app exits. If you need post_run, set exec_mode = "fork" in your TOML.

Using shell features in hooks

Hooks don't go through a shell. If you need pipes, redirects, or variable expansion, wrap in sh -c:

toml
[hooks]
pre_run = ["sh -c 'echo Starting app > /tmp/myapp.log 2>&1'"]
📚 Academic
Hooks use a declarative array-of-strings model. Each string is split on whitespace (with basic quote handling) by hooks.splitCmd() and passed directly to exec.Command() — no shell invocation. This keeps the execution surface small and predictable: no injection vectors from user-controlled data, no PATH-dependent shell behavior. The trade-off is that shell features require explicit sh -c '...' wrapping, which is visible and auditable in the TOML.

Advanced: exec vs fork mode

The exec_mode setting controls how the stub hands off to your app.

ModeHowpost_run works?PIDWindows

Grug use exec (default). Stub become app. Same process. Clean. No wrapper overhead. If grug need to run cleanup script after app exits, grug use fork. Otherwise, grug leave it alone.

Advanced: Persist Extraction

By default, the stub extracts to a fresh temp directory on every run. With persist = true, it extracts once to ~/.bindboss// and reuses that directory on subsequent runs — extraction is skipped if the directory already has content.

toml
[extract]
persist = true   # extract once, reuse forever

Or via CLI:

shell
bindboss pack ./myapp myapp-bin --run="julia main.jl" --persist

Use persist for apps with large runtimes (Julia, Node.js deps, ML models) where extraction every run would be slow. For small scripts, the default tmpdir behavior is fine.

✓ Tip
To reset the persistent extraction (e.g. after updating the binary), just delete ~/.bindboss// or use bindboss reset which clears the state file and forces re-extraction on next run.

Advanced: Environment Variables

You can inject extra environment variables into your app's environment using the env field in bindboss.toml:

toml
env = [
    "APP_ENV=production",
    "LOG_LEVEL=info",
    "MAX_CONNECTIONS=100"
]

These are appended to the process environment (on top of the existing env). If the same key already exists in the process env, the last value wins — so your TOML values take priority over the system environment.

The stub also sets one implicit env var: users can set BINDBOSS_VERIFY=1 before running to force hash verification. This is external — you don't configure it in TOML.

Advanced: Signing & Verification

Signing adds an Ed25519 signature over the payload hash. This lets users verify that a binary was produced by you and has not been tampered with since signing.

Full workflow

Example 28 Producer side (you) 
# Generate once
bindboss keygen myproject
# ~/.bindboss/keys/myproject.key  (KEEP SECRET)
# ~/.bindboss/keys/myproject.pub  (share this)

# Pack with signing
bindboss pack ./myapp myapp-v1.2 \
    --run="python3 main.py" \
    --sign=~/.bindboss/keys/myproject.key

# Distribute: myapp-v1.2 + myproject.pub
Example 29 Consumer side (your user) 
# They run verify before running
bindboss verify ./myapp-v1.2 --pubkey=./myproject.pub
# [bindboss] ✓ hash verified
# [bindboss] ✓ signature verified (Ed25519)

./myapp-v1.2
📚 Academic
The Ed25519 signature is computed over SHA-256(payload) rather than the raw payload bytes. Signing a cryptographic hash is safe when the hash function is collision-resistant: SHA-256 provides 128-bit collision resistance, which is well above the 80-bit threshold required for Ed25519 security. This approach avoids loading the entire (potentially large) payload into memory for signing — only the 32-byte hash is signed. The signature is stored in the 64-byte slot in the v2 trailer; if no signing key was provided at pack time, those 64 bytes are zeros and the sig_present flag bit is 0.

Advanced: Go Library API

bindboss exposes a Go library API at pkg/bindboss for embedding packing functionality into your own tools.

go
import "github.com/marshalldavidson61-arch/bindboss/pkg/bindboss"

func packMyApp() error {
    opts := bindboss.PackOptions{
        SrcDir:  "./myapp",
        OutPath: "myapp-bin",
        Run:     "python3 main.py",
        Persist: true,
    }
    return bindboss.Pack(opts)
}

See the pkg/bindboss/bindboss.go source for the full options struct and available functions.

!!! FATAL: no run command specifiedNo run in bindboss.toml and no --run flag. One of them must be set.
