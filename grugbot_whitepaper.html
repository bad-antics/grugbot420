1. Overview & Philosophy

GrugBot is a neuromorphic AI engine written in Julia that models cognition through competing node populations rather than traditional neural networks. Each node is an independent pattern-matcher that votes for actions based on input similarity. The system exhibits emergent behavior through stochastic competition, strength-based natural selection, and cross-domain cascade activation across specialized brain lobes.

Core Design Principles

  • No Silent Failures — Every error is surfaced loudly. All catch blocks log via @warn or rethrow. Zero bare catch blocks.
  • Stochastic Competition — Nodes compete via weighted coinflips, not deterministic ranking. The @coinflip macro drives all probabilistic decisions.
  • Biological Metaphors — Lobes partition knowledge domains. BrainStem dispatches signals. Phagy cycles clean dead nodes. The prefrontal cortex (AIML) orchestrates cross-domain reasoning.
  • GRUG Comment Style — All code uses inline cave-speak comments (# GRUG:) plus academic footer blocks for formal documentation.
  • Thread Safety — All shared state protected by ReentrantLock. Node map, queues, and thesauri are thread-safe.
  • Ambient Field / Instant Activation — The node population is never fully off. The signal is not transmitted — it is already present. Intensity rise is instant because the field is pre-aligned. See §17.5.
The architecture draws from Hopfield networks (associative memory caching), competitive learning (strength-biased selection), apoptosis (grave markers for failed nodes), and winner-take-all lateral inhibition (BrainStem dispatch). The AIML orchestrator functions as a prefrontal cortex — a centralized executive that integrates votes from all specialized cortical regions (lobes) into a unified response. The always-on ambient field and resolution-latency model (§17.5) mean that familiar inputs resolve near-instantly via Hopfield cache, while novel inputs trigger full stochastic competition — mirroring biological attention and rest-state consolidation.

3. Nodes: The Atomic Unit

Every piece of knowledge in GrugBot is stored as a Node — a mutable struct with a text pattern, action packet, strength score, neighbor links, and metadata.

Node Structure

mutable struct Node
    id::String                    # Unique ID (node_0, node_1, ...)
    pattern::String               # Tokenized text pattern for matching
    action_packet::String         # Pipe-delimited actions with weights + negatives
    json_data::Dict{String,Any}   # Arbitrary metadata (system_prompt, etc.)
    neighbor_ids::Vector{String}  # Linked nodes (max 4 → UNLINKABLE)
    strength::Float64             # 0.0 – 10.0 (STRENGTH_CAP)
    is_grave::Bool                # Dead node (negative reinforcement anchor)
    grave_reason::String          # Why it died (GRAVED-SLOW, strength→0, etc.)
    is_unlinkable::Bool           # At max neighbors (4)
    hopfield_key::UInt64          # Associative memory hash
    relational_patterns::Vector{RelationalTriple}
    response_times::Vector{Float64}
    is_image_node::Bool           # True if pattern is image SDF data
end

Node Lifecycle

EventEffect
CreationStrength = 1.0, hopfield key computed, auto-latch neighbors (if >1000 nodes)
Vote castbump_strength! on coinflip → strength increases toward cap
/wrongpenalize_strength! on coinflip → strength decreases
Strength → 0Marked is_grave = true, becomes negative anchor
Slow responseAvg time > 5s → GRAVED-SLOW
4 neighborsis_unlinkable = true, rejects further links

Strength-Biased Selection

When scanning, each alive node gets a coinflip weighted by its strength:

P(activate) = 0.1 + 0.8 × (strength / STRENGTH_CAP)

Strength 0.0 → 10% chance. Strength 10.0 → 90% chance. This creates Darwinian selection pressure.

4. Action Packets v2 (Pipe-Delimited) v7

Action packets define what a node does when it wins a vote. Version 2 uses pipe-delimited format with inline negatives per action, allowing action names to contain commas.

Format

# Single action, default weight (1.0)
"greet"

# Single action with weight
"reason^3.5"

# Single action with inline negatives and weight
"greet[dont frown, dont insult]^2"

# Multiple actions, pipe-delimited
"greet^3 | reason^2 | analyze^1"

# Full format: pipe-delimited with mixed negatives
"greet[dont frown, dont insult]^3 | welcome[dont be rude]^2 | smile^1"

# Action names CAN contain commas (pipes are the delimiter)
"plan, execute^2 | greet^1"

Parser Return Values

function parse_action_packet(packet::String)
    → (positives, all_negatives, action_items)

    positives    :: Vector{Tuple{String, Float64}}
    # (action_name, weight) pairs for weighted selection

    all_negatives :: Vector{String}
    # Deduped union of ALL negatives across all actions

    action_items :: Vector{Tuple{String, Float64, Vector{String}}}
    # (action_name, weight, per_action_negatives) — full detail
end

Weighted Selection

select_action(packet) normalizes weights to probabilities and rolls a Categorical distribution via @coinflip:

P(action_i) = weight_i / Σ(weights)

For "heavy^10 | light^1": P(heavy) ≈ 91%, P(light) ≈ 9%

Inline Negatives

Negatives tell Grug what NOT to do. Each action can have its own "dont" list inside square brackets. When greet says [dont frown, dont insult], those are constraints passed to the AIML generator so the response avoids those behaviors. All negatives from all actions are also collected into a flat list for backward compatibility with the vote system.
⚠ Migration from v3: Old comma-delimited format "reason^2,analyze^1" is no longer valid. Use pipe-delimited: "reason^2 | analyze^1"

5. Lobes: Subject-Specific Partitions

Lobes partition the node map into specialized cognitive regions. Each lobe has a subject description, a node capacity (default 20,000), fire/inhibit counters, and bidirectional connections to other lobes.

Lobe Structure

mutable struct LobeRecord
    id::String               # Unique lobe ID
    subject::String          # Domain description
    node_ids::Set{String}    # Nodes assigned to this lobe
    connections::Set{String}  # Connected lobe IDs (bidirectional)
    node_cap::Int            # Max nodes (default 20,000)
    fire_count::Int          # Times this lobe activated
    inhibit_count::Int       # Times this lobe was suppressed
end

Cross-Lobe Cascade

When nodes fire in a primary lobe, scan_and_expand() performs two expansion passes:

PassSourceConfidenceDescription
1. Drop-TableSame lobe neighbors80% of activatorCo-activation of linked nodes within the same domain
2. Lobe CascadeOther active lobes60% of max primaryCross-domain bridge activation (threshold: 0.15 min confidence)
Lobe cascade is like Grug hearing a word in science cave, then peeking into philosophy cave to see if anyone there has something to say too. Philosophy cave nodes get quieter voice (60% confidence) but they still vote! This is why cross-domain queries work.

7. BrainStem: Winner-Take-All Dispatch

The BrainStem implements lateral inhibition across lobes. When multiple lobes produce competing responses, the BrainStem selects a winner based on confidence and propagates inhibitory signals to losers.

Dispatch Flow

  1. Each lobe's top-scoring node submits a bid (confidence score)
  2. BrainStem selects highest confidence bid as winner
  3. Winner's lobe gets a propagation signal (reinforcement)
  4. Losing lobes get inhibitory feedback
This models the thalamo-cortical loop in biological brains where the thalamus (BrainStem) gates which cortical region (lobe) gets to dominate the current processing cycle. The inhibitory propagation prevents runaway excitation and ensures focused responses.

8. Prefrontal Cortex: AIML Orchestration v7

The AIML orchestrator is GrugBot's prefrontal cortex — a centralized executive that integrates votes from ALL lobes into a unified response. It is NOT per-lobe; it sees the entire cognitive state.

Orchestration Pipeline

  1. Vote Collection: All specimens from all lobes cast votes
  2. Confidence Sort: Votes sorted by confidence descending (highest-confidence domain node wins primary)
  3. Bucketing: Sure votes (within 0.05 of max) vs. Unsure votes (50/50 coinflip)
  4. Lobe Context: extract_lobe_aware_context(votes) maps votes → lobes, builds cross-domain awareness
  5. Rule Evaluation: Stochastic AIML rules fire based on their probability, with template variables expanded
  6. Response Generation: JIT text synthesis combining all context layers

AIML Template Variables

VariableExpands To
{MISSION}Original user input text
{PRIMARY_ACTION}Winning vote's action (e.g., "reason")
{SURE_ACTIONS}Comma-separated list of high-confidence actions
{UNSURE_ACTIONS}Coinflip-surviving lower-confidence actions
{ALL_ACTIONS}Every action from every vote
{CONFIDENCE}Primary vote's confidence score
{NODE_ID}Primary vote's node ID
{MEMORY}Pinned + recent message history
{LOBE_CONTEXT} v7Active lobes with node counts and sample patterns

Lobe-Aware Context

The extract_lobe_aware_context(votes) function maps all votes to their owning lobes and builds a context string showing which brain regions are active:

# Example output:
Lobe Context: [philosophy (2/4 active: truth knowledge logic | ethical moral)]
            | [science (1/3 active: hypothesis experiment observe)]
            | [reasoning (1/2 active: deduce infer analyze)]
Before v6, AIML could only see the vote stream. If you asked about "hypothesis truth", the science lobe fired and philosophy was a quiet cascade vote at 60%. Now the prefrontal cortex SEES both lobes explicitly via {LOBE_CONTEXT}, so AIML rules can reason across domains: "I see science AND philosophy are active — integrate both!"

The Thesaurus ships with 40+ seed synonym groups across domains:

  • Science (10): hypothesis, experiment, observe, analyze, energy, matter, evolve, cause, effect, measure

Before pattern scanning, the Thesaurus gate expands input tokens with known synonyms. If "happy" is in the input, "joyful", "glad", "cheerful" are added to the scan set, increasing match probability against nodes that use different but semantically equivalent words.

11. Phagy Mode: Self-Maintenance

Phagy cycles run during idle periods (triggered by a 50/50 coinflip when the cave has been quiet for ~120s ±30s, requires 1000+ alive non-image nodes). One of six maintenance automata runs per cycle:

#AutomatonDescription
1ORPHAN_PRUNERFind nodes with 0 neighbors and no recent activity → grave them
2STRENGTH_DECAYERGlobally decay node strengths slightly (use-it-or-lose-it)
3GRAVE_RECYCLERRemove very old grave nodes to free memory
4CACHE_VALIDATORAudit Hopfield cache for stale entries
5DROP_TABLE_COMPACTRemove broken links in drop tables
6RULE_PRUNERRemove AIML rules that reference graved nodes
Only ONE automaton runs per cycle. This is how Grug keeps Big-O safe. Seven automata running at once would be like six cave bears in one cave. Bad!

12. EyeSystem: Visual Processing

The EyeSystem handles image inputs through a biologically-inspired visual pipeline:

  1. Image Detection: detect_image_binary() identifies Base64 image data URIs
  2. SDF Conversion: Image pixels → SDFParams (x, y, brightness, color arrays)
  3. Edge Blur: Gaussian blur simulating peripheral vision degradation
  4. Attention Map: Arousal-gated spatial attention with centroid weighting
  5. Arousal Modulation: Global arousal level (0.0-1.0) gates visual sensitivity
  6. Jitter: Stochastic noise injection for robust matching

13. ChatterMode: Gossip Sessions v7.1

Chatter sessions simulate "dreaming" — 50–500 node clones are placed in a gossip pool where weak nodes drift toward strong neighbors. This creates directional knowledge flow: strong nodes signal, weak nodes learn. Chatter only fires for mature specimens with 1000+ nodes. New specimens do not chatter.

Population Gate

Chatter requires a minimum of 1000 alive non-image nodes. Below this threshold, the specimen is too young for idle gossip — topology is still under direct user control via /grow and drop_table wiring. Chatter would destabilize immature specimens.

Idle Timer (Shared with Phagy)

Both chatter and phagy share the same slow idle timer: 120 seconds default, ±30s jitter band. Idle events fire between 90s and 150s apart. A 50/50 coinflip decides chatter vs phagy. This is much slower than v7 (was 30s ±5s).

Group Size

Each chatter round selects 50–500 nodes (was 100–800 in v7). If fewer than 50 eligible nodes exist, the floor is set to whatever is available (use entire population as ceiling).

Weak-Only Morph

Only receivers strictly weaker than the sender can accept a pattern blend. Strong nodes signal but never change themselves. This creates directional knowledge flow: proven nodes teach, unproven nodes learn. Receiver strength is compared using the original un-jittered value, not the chatter-jittered value.

Once-Per-Day Morph Limit

Each node can only morph once every 24 hours (86,400 seconds). Tracked via MORPH_COOLDOWN_MAP (node_id → last_morph_timestamp). This prevents runaway drift where a weak node gets blended every chatter round and loses its original identity entirely. After morphing, the node is locked for 24 hours before it can morph again.

Gossip Rules

  • Anti-collision: nodes can\'t exchange with themselves or repeat exchanges in one round
  • Sender coinflip: 50/50 chance to initiate gossip per clone
  • Strength gate: receiver must be weaker than sender (original strength)
  • Cooldown gate: receiver must not have morphed in the last 24 hours
  • Biased copy: copy probability [0.3, 0.8] scales with sender strength
  • Pattern blending: tokens from sender pattern are injected into receiver (blend factor [0.1, 0.4])
  • Action packet blending: vote slots are slightly mixed (0.5× blend factor)
  • User input during chatter: queued via INPUT_QUEUE, drained after session completion
v7.1 Chatter Summary: Idle timer 120s ±30s (shared with phagy). Population gate: 1000+ nodes. Group size: 50–500. Weak-only morph with 24h cooldown. Strong nodes signal, weak nodes learn. Directional knowledge flow prevents strong-node drift and weak-node identity loss.

15. ActionTonePredictor: Input Classification

Classifies user input into action families and tone families before scanning:

Action Families

  • ACTION_QUERY — questions ("what", "why", "how")
  • ACTION_COMMAND — directives ("run", "do", "execute")
  • ACTION_ASSERT — statements of fact

Tone Families

  • TONE_NEUTRAL — default
  • TONE_CURIOUS — exploratory questions
  • TONE_HOSTILE — aggressive/negative input
  • TONE_URGENT — time-critical, escalated
  • TONE_REFLECTIVE — thoughtful, hedged

Arousal Nudge

The predictor calculates an arousal nudge that's applied to EyeSystem:

  • Hostile/Urgent → positive nudge (heightened arousal)
  • Reflective → negative nudge (calmed arousal)
  • Neutral/Curious → no nudge

Weight Multiplier

Nodes whose action aligns with the predicted action type get a confidence multiplier >1.0; misaligned nodes get <1.0. This biases voting toward contextually appropriate responses.

16.5 Relational Fire System v7.2

The relational fire system introduces user-defined relay circuitry into the node engine. Where drop tables provide implicit co-activation (neighbors fire when the primary fires), and lobe cascades provide cross-domain propagation (decayed signals hop between connected lobes), the attachment system provides explicit, targeted, pattern-carrying relay links between specific nodes.

Architecture

The system is built on four primitives stored in engine.jl:

  • AttachedNode — immutable struct: node_id::String, pattern::String (connector/middleman pattern), signal::Vector{Float64} (pre-baked via words_to_signal)
  • ATTACHMENT_MAP — Dict{String, Vector{AttachedNode}} mapping target node IDs to their attached nodes
  • ATTACHMENT_LOCK — ReentrantLock for thread-safe access
  • MAX_ATTACHMENTS = 4 — hard cap per target node

Attachment Lifecycle

Attach: /nodeAttach [ ...]

Validation is exhaustive and non-silent: empty arguments, node existence, grave status, self-attach prevention, max cap enforcement, and duplicate rejection all produce explicit error messages. The pattern is pre-baked into a signal vector at attach time so no allocation occurs during the hot scan path.

Detach: /nodeDetach removes a specific attachment. If that was the last attachment on the target, the entry is cleaned up entirely.

Inspect: /attachments prints every target and its attached nodes with patterns, signal vector lengths, and slot usage (N/4).

Firing Mechanics (Pass 3 of scan_and_expand)

The attachment relay runs as Pass 3 in the scan_and_expand() pipeline, after the primary scan (Pass 1) and lobe cascade (Pass 2). For every node in the expanded set, the engine checks ATTACHMENT_MAP for attached nodes. Each attachment goes through a three-stage gate:

  1. Active cap gate: If the relay has already hit its independent cap (rand(600:1800)), remaining attachments are skipped. This respects the biological attention bottleneck.
  2. Alive check: If the attached node has been deleted or graved since attachment, it is silently skipped (stale attachment, non-fatal).
  3. Strength-biased coinflip: scan_prob = 0.20 + (strength / STRENGTH_CAP) × 0.70. Strong attachments fire more often. Weak ones still have a 20% floor.

Winners that pass all three gates get their connector pattern (middleman) scanned against the attached node's own pattern — not the target's. The connector pattern represents WHY these nodes are related. Confidence is determined by how well the relay reason matches the waking node:

base_conf = token_overlap_similarity(connector_pattern, attached_node_pattern)
strength_bonus = attachment_strength / STRENGTH_CAP × 0.5
jitter = randn() × RELAY_CONF_JITTER_SIGMA   (sigma = 0.05)
confidence = max(0.1, base_conf + strength_bonus + jitter)

The confidence floor of 0.1 ensures that attached nodes always have some voice in the quorum, even if the pattern overlap is minimal. The small stochastic jitter (sigma=0.05) is biologically motivated — synaptic relay is inherently noisy, and the same attached node should not fire with identical confidence every cycle. This nudges vote pool diversity without dominating the signal. The connector pattern also surfaces downstream as a RelationalTriple(target_id, "relay_attached", connector_pattern) so the generative pipeline knows WHY the relay fired. Fired attachments receive a bump_strength! call — they earned usage credit. Deduplication ensures no node appears twice in the expanded set.

Specimen Persistence

Attachments are fully serialized as section 14 of /saveSpecimen and restored as section 4.14 of /loadSpecimen. Each entry stores target_id, node_id, pattern, and the pre-baked signal vector. On load, if the signal vector is missing (backward compatibility with older specimens), it is re-baked from the pattern via words_to_signal.

Design Rationale

The attachment system is the explicit counterpart to the engine's implicit co-activation mechanisms. Drop tables discover neighbors through seeding; lobe cascades propagate through connectivity. Attachments let the architect declare: "when this node fires, give these specific nodes a chance to contribute with this specific framing." The coinflip gate ensures that explicit wiring does not bypass the stochastic selection pressure that keeps the system honest. The biological parallel is pre-wired axonal pathways — genetically encoded relay circuits that still require synaptic gating to fire.

17. Formal Mathematics

Strength Dynamics

Bump (on vote):

s(t+1) = min(s(t) + δ, STRENGTH_CAP) where δ ~ Bernoulli(0.5)

Penalize (/wrong):

s(t+1) = max(s(t) - δ, 0.0) where δ ~ Bernoulli(0.5)

Grave condition: s(t) = 0.0 → is_grave = true

Scan Confidence

Token overlap:

conf_token = |input_tokens ∩ node_tokens| / max(|input_tokens|, |node_tokens|)

Relational triple boost:

conf_rel = Σ(matching_triples) × TRIPLE_WEIGHT

Combined:

conf = conf_token × TOKEN_WEIGHT + conf_rel × REL_WEIGHT

Cascade Confidence

Drop-table (same lobe):

conf_drop = conf_activator × 0.8

Lobe cascade (cross-lobe):

conf_cascade = max(conf_primary) × 0.6 (only fires if conf_cascade ≥ 0.15)

Action Selection

Weighted random:

P(action_i) = weight_i / Σ(all_weights)

Implemented via Categorical distribution from Distributions.jl, wrapped in @coinflip macro.

17.5 Signal Presence & Instant Activation v7

The most important performance insight in GrugBot is not about throughput or parameter count. It is about the nature of the signal itself. In transmission-based architectures, a signal races from input to output through a series of discrete steps: tokenize → embed → attend → decode. Latency is the sum of those transmission costs. GrugBot operates differently: the signal does not race because it is already present.

Always-On Ambient Field

The node population is never fully off. Between interactions the system runs at ambient low flow: ChatterMode gossip, PhagyMode maintenance, Hopfield cache warm. When input arrives and intensity rises, nothing is assembled from scratch. The activation is not a signal traveling through the network — it is a state change in a field that was already present and already aligned. Intensity rise is therefore instant: there is no cold-start propagation delay because there is no propagation.

Transmission latency model (conventional):

Ltotal = Ltokenize + Lembed + Lattend + Ldecode

Resolution latency model (GrugBot):

Ltotal = Lcollapse = time for pre-loaded field to reach quorum agreement

When Hopfield cache hits: Ltotal ≈ O(1) — cache lookup only, no scan

When quorum is sure (nodes within 0.05 of peak confidence): fast collapse

When quorum is unsure: full stochastic scan required, collapse is slower

The Inverse Mode: Low-Flow Activation

The same principle operates in reverse. At low ambient flow — when no input is present — the system does not go quiet. It activates in a different mode: ChatterMode. Ephemeral clones of 50–500 nodes exchange patterns on a coinflip (1000+ node population required). No external input is required. The activation is driven by the internal field state. The network restructures itself along lines of internal coherence — consolidating associations, cross-linking related patterns, pruning irrelevant connections. The low-flow state is not a deficit. It is the system maintaining the pre-alignment that makes instant activation possible.

The Entanglement Principle

Quantum entanglement is not faster-than-light communication. Two entangled particles do not exchange a signal when one is measured. The correlation was established at the moment of entanglement. The measurement does not send anything — it reveals a pre-existing relational state.

GrugBot's always-on field operates on the same structural principle. When input arrives and a node fires at high confidence, it is not because the signal raced through the network and found a match. It is because the node was already aligned with that pattern domain. By the time the strength-biased coinflip fires and the node is scanned, every factor determining the outcome has already been established:

  • Pattern vector pre-loaded at node creation
  • Strength pre-accumulated through prior interactions and /wrong feedback
  • Lobe membership pre-established through domain assignment
  • Hopfield cache pre-recorded if this input hash has been seen before
  • ActionTonePredictor pre-shaped the scan field before the first node was reached
  • EyeSystem arousal pre-set the attention gate width
  • NegativeThesaurus pre-stripped inhibited tokens from the input

The scan is not a computation — it is a measurement of a pre-existing state. The response speed is therefore not a function of how fast the computation runs. It is a function of how well the field was pre-aligned.

Formal statement of the resolution latency principle:

Let F(t) be the field state at time t (node strengths, Hopfield cache, lobe topology, arousal baseline). Let I be an input. The response R is determined entirely by F(t) and I. The latency L is the time required for the quorum defined by F(t) to collapse under I. When F(t) is well-aligned to I (high-confidence sure quorum, Hopfield hit), L → 0. When F(t) is poorly aligned (weak unsure quorum, no cache), L is bounded by the full stochastic scan cost.

The field is always loaded. The only question is how well it is aligned to the current input.

Quorum Settlement as Collapse Speed

The sure/unsure quorum split is not only an uncertainty representation — it is a direct measure of field alignment quality. A sure quorum that settles fast (multiple nodes landing within 0.05 of the same peak confidence) means the field was well-aligned to this input. An unsure quorum that requires the full coinflip population means the field was only partially aligned. The collapse took longer not because the system is slower, but because pre-existing alignment was weaker.

Sure quorum condition:

sure = { v ∈ votes : v.confidence ≥ max_conf - 0.05 }

Alignment quality metric:

alignment_quality = |sure| / |votes_total|

High ratio → field well-aligned to input → fast resolution
Low ratio → field weakly aligned → slow resolution, unsure quorum dominant

Hardware Correspondence

The same principle governs the Pocket Antikythera hardware architecture. A mechanical system with pre-loaded tension — spring-loaded gear trains, pre-stressed couplings within their tolerance bands — does not transmit force across a distance when triggered. The energy is already present in the mechanical field. The trigger releases a pre-existing alignment. The response is instant not because the trigger is fast but because the energy was already there, the gears already meshed, the slack already calibrated.

Hall effect sensors, optical sensors, and strain gauges in such a system do not detect an incoming signal. They detect the current state of a pre-existing mechanical field. The quorum alignments in the digital feedback layer do not aggregate transmitted votes — they measure the current agreement state of a field that was already running.

Software and hardware, the principle is identical: pre-alignment eliminates transmission. The signal is not in the wire. Only the computation is. And the computation was already running.
